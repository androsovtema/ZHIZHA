# ЖИЖА — полный UX-аудит и инвентаризация текущего продукта

Дата аудита: 2026-06-22  
Цель документа: дать Claude Design фактическую карту текущей платформы ЖИЖА перед полным UX-редизайном. Это не новый дизайн и не список визуальных вкусовых правок. Документ описывает, как продукт сейчас устроен, что пользователь видит, какие сценарии доступны, какие настройки существуют, где есть UX-дыры и какие нюансы важно сохранить или переосмыслить.

## 1. Что такое ЖИЖА сейчас

ЖИЖА — статическое фронтенд-приложение для генерации процедурных живых фонов. Вся основная платформа находится в одном файле `index.html`: UI, стили, WebGL/Three.js-движок, GLSL-шейдер, пресеты, шаблоны, экспорт, запись видео, URL-шэринг и embed-режим.

Основная пользовательская ценность сейчас:

- быстро выбрать назначение фона;
- выбрать живой шаблон;
- подстроить цвет, скорость, движение, детализацию и эффект;
- скачать PNG или видео;
- при необходимости получить ссылку/iframe для сайта.

Техническая ценность:

- процедурный WebGL-фон без бэкенда;
- статический деплой на GitHub Pages;
- сцена кодируется в URL;
- iframe/embed-режим скрывает UI;
- отдельный веб-компонент `<zhizha-bg>` позволяет использовать фон на чужом сайте.

## 3. Структура проекта

Проект очень компактный.

```text
/
  index.html
  README.md
  assets/
    icon-180.png
    og.jpg
  backups/
    index.v1.backup.html
  docs/
    BRAND_KIT.md
    CREATOR_UI_IMPLEMENTATION_PROMPT.md
    ux-audit/
      ZHIZHA_UX_INVENTORY.md
      screenshots/
  embed/
    EMBED.md
    zhizha-bg.js
```

Текущая архитектура:

- Нет сборки, npm-приложения, роутинга, бэкенда, аккаунтов, базы данных.
- Three.js подключается через import map с CDN: `https://unpkg.com/three@0.160.0/build/three.module.js`.
- Google Fonts подключаются напрямую: Comfortaa, Nunito, JetBrains Mono.
- Все состояния живут в JS-объекте `state`.
- Сцена сериализуется в URL hash `#s=...`.
- Тема UI и ширина панели сохраняются в `localStorage`.

## 4. Главные поверхности продукта

### 4.1. Основное приложение

Основное приложение состоит из двух колонок на desktop:

- левая колонка: панель управления;
- правая колонка: canvas preview.

На mobile:

- preview закреплен сверху через sticky-зону;
- панель управления идет ниже как длинный лист.

### 4.2. Embed-режим

Embed-режим включается, если URL содержит `embed=1`. В этом состоянии:

- скрываются панель, live bar, FPS, кнопка панели и индикатор записи;
- stage занимает весь viewport;
- canvas становится full-bleed без скругления и тени.

Важный нюанс: если на уже загруженной странице просто поменять hash на `#embed=1`, режим не включится, потому что код добавляет класс `body.embed` только в `init()` при загрузке. При прямом открытии `?embed=1` или новой загрузке URL режим работает.

### 4.3. Веб-компонент для сайта

Файл `embed/zhizha-bg.js` регистрирует custom element `<zhizha-bg>`.

Атрибуты:

- `src` — URL сцены ЖИЖА, желательно с `embed=1`;
- `fallback` — статичная картинка на случай, если iframe не загрузится;
- `z` — z-index слоя, по умолчанию `-1`.

Поведение:

- компонент создает shadow DOM;
- рисует fixed full-screen слой;
- ставит `pointer-events:none`, чтобы фон не перехватывал клики;
- может показать fallback-изображение под iframe;
- iframe получает `title="Фон"`, `loading="lazy"`, `tabindex="-1"`, `aria-hidden="true"`.

## 5. Текущий layout и визуальная логика

### Desktop

Body — CSS grid:

- `grid-template-columns: var(--panel-w, 500px) 1fr`.
- Панель слева, canvas справа.
- Ширина панели регулируется ручкой `#panelResize`, диапазон 340-820 px.
- Выбранная ширина сохраняется в `localStorage` как `zhizha_panel_w`.

Canvas:

- расположен внутри `#stage > #stageInner > #canvasWrap`;
- сохраняет выбранное соотношение сторон;
- не перекрывается панелью;
- имеет скругление, тень и черный фон;
- в embed-режиме занимает весь viewport.

Панель:

- полупрозрачное матовое стекло;
- прокручивается отдельно от preview;
- содержит бренд, переключатель режима, блоки настроек, экспорт, автора.

### Mobile

Breakpoint: `max-width: 760px`.

Изменения:

- body становится обычным block layout;
- stage становится sticky сверху, высота `46vh`;
- панель становится статичной нижней областью;
- preview остается видимым при скролле;
- ручка изменения ширины панели скрыта;
- FPS скрыт;
- panel toggle фактически не используется.

UX-нюанс: на mobile пользователь сначала видит preview и начало creator-панели, но быстрые настройки и экспорт находятся сильно ниже из-за длинной сетки шаблонов. Для TikTok/Reels в категории доступно 38 шаблонов, поэтому до быстрых настроек нужно долго скроллить.

## 6. Брендовая и системная оболочка UI

### Брендовый блок

Вверху панели:

- логотип-квадрат с буквой `Ж`;
- название `ЖИЖА`;
- подзаголовок `живые фоны за 20 секунд`;
- кнопка выбора темы оформления.

### Темы оформления

Тема — это настройка UI-оболочки, не палитры генерируемого фона.

Доступные темы:

- `Вольт` — тема по умолчанию;
- `Карамель`;
- `Магма`;
- `Хром`.

Поведение:

- тема хранится в `localStorage` как `zhz-theme`;
- ранний скрипт в `<head>` поднимает сохраненную тему до первой отрисовки, чтобы не было мигания;
- меняются CSS variables: фон, панель, текст, акценты, градиент action-элементов;
- форма, layout и поведение не меняются.

UX-риск: визуально выбор темы похож на выбор палитры фона. Это две разные сущности, но в UI обе представлены цветными свотчами/градиентами, поэтому пользователь может не сразу понять, что тема меняет интерфейс, а не результат.

## 7. Режимы продукта

### 7.1. Простой режим

Это стартовый режим. `body` получает класс `mode-simple`.

Виден:

- creator-блок `Для чего нужен фон?`;
- выбор стиля;
- быстрые настройки;
- экспорт;
- автор.

Скрыто:

- точный размер;
- поле;
- фактура;
- расширенные цвета;
- блок для сайта.

Стартовое поведение:

- если в URL нет `#s=...`, приложение выбирает первую creator-категорию: `TikTok / Reels`;
- категория ставит ratio `9:16`;
- применяется дефолтный шаблон категории: `Вспышка`;
- preview сразу становится вертикальным.

### 7.2. Профи режим

Включается кнопкой `⚙ Профи`. `body` получает класс `mode-pro`.

Виден:

- размер;
- поле;
- фактура;
- цвета;
- экспорт;
- блок `Для сайта · ссылка и встраивание`;
- автор.

Скрыто:

- creator-категории;
- template cards;
- быстрые настройки.

Pro — это техническая поверхность для точной настройки процедурной сцены. Она раскрывает внутреннюю модель: ratio, field, speed, scale, warp, grain, surface, pixel, halftone, custom color stops, tile count.

UX-нюанс: режимы переключаются как равноправные вкладки, хотя по сути это разные уровни сложности. При переходе в Pro пользователь видит сохраненную сцену из Simple, но теряет creator-контекст: выбранная категория и шаблон больше не видны.

## 8. Текущая информационная архитектура

Порядок секций задается через flex `order`:

1. Бренд.
2. Переключатель `Простой / Профи`.
3. В Simple: creator step.
4. В Simple: quick step.
5. Экспорт.
6. В Pro: размер.
7. В Pro: поле.
8. В Pro: фактура.
9. В Pro: цвета.
10. В Pro: для сайта.
11. Автор.

Фактически из-за `order` экспорт находится после quick step в Simple и после Pro-настроек в Pro.

Главная проблема текущей IA: экспорт является главным действием, но не всегда находится рядом с моментом принятия решения. На desktop он виден после скролла панели. На mobile он находится далеко ниже ленты шаблонов.

## 9. Creator-сценарий в Simple

Текущий главный путь:

```text
Открыть приложение
-> автоматически выбран TikTok / Reels
-> выбрать категорию назначения
-> выбрать mood-фильтр
-> выбрать шаблон
-> настроить цвет / скорость / движение / детализацию / эффект
-> выбрать разрешение
-> скачать PNG или записать видео
```

### Категории назначения

Всего 8 категорий:

| Категория | Ratio | Дефолтный шаблон | Доступные семьи | Кол-во шаблонов |
|---|---:|---|---|---:|
| TikTok / Reels | 9:16 | Вспышка | energy, deep, ornament, graphic | 38 |
| Stories | 9:16 | Шёлк | calm, deep, graphic, mono | 33 |
| YouTube | 16:9 | Аврора | deep, graphic, energy, calm, mono | 43 |
| Фон под текст | 16:9 | Затишье | calm, deep, mono | 23 |
| Обложка | 4:5 | Фейерверк | energy, ornament, graphic, deep, mono | 43 |
| Презентация | 16:9 | Туман | calm, deep, graphic, mono | 33 |
| Паттерн | 1:1 | Соты | graphic, ornament, mono | 24 |
| Абстрактный луп | 16:9 | Галактика | energy, deep, ornament | 28 |

Категория делает две вещи:

- меняет формат preview;
- фильтрует доступные шаблоны по семействам.

Категория не меняет экспортную кнопку напрямую, но `catHint` показывает подсказку: например, для TikTok/Reels — `Вертикальное видео — экспорт видео на первом месте.`

UX-дыра: подсказка говорит, что видео на первом месте, но в блоке экспорта первой кнопкой все равно идет `Скачать PNG`, затем запись видео.

### Mood-фильтры

Фильтры формируются из семейств выбранной категории:

- `Все`;
- `Спокойные`;
- `Яркие`;
- `Глубокие`;
- `Графика`;
- `Узоры`;
- `Монохром`.

Не все фильтры есть в каждой категории. Фильтр влияет только на список шаблонов, а не на текущую сцену сам по себе.

### Шаблоны

Текущий UI показывает 52 creator-шаблона. Каждый шаблон задает полный набор процедурных значений:

- speed;
- scale;
- warp;
- grain;
- field;
- palette;
- seamless;
- screen;
- dots;
- pixel;
- dot.

Семьи шаблонов:

- calm: 9 шаблонов;
- energy: 10;
- deep: 9;
- graphic: 10;
- ornament: 9;
- mono: 5.

Полный список:

- calm: Туман, Шёлк, Дымка, Молоко, Пастель, Затишье, Вуаль, Перо, Роса.
- energy: Вспышка, Кислота, Разряд, Пульс, Жар, Стробоскоп, Фейерверк, Турбо, Рейв, Лава.
- deep: Бездна, Галактика, Глубина, Аврора, Эфир, Омут, Мираж, Млечность, Космос.
- graphic: Пиксель, Соты, Терминал, Полутон, Решётка, Контур, Растр, Микросхема, Ткань, Мозаика.
- ornament: Мандала, Калейдоскоп, Подсолнух, Розетка, Спираль, Снежинка, Витраж, Орнамент, Лотос.
- mono: Графит, Тушь, Сталь, Дым, Нуар.

Шаблон визуально представлен карточкой:

- квадратная карточка;
- live preview canvas внутри;
- градиентный fallback-фон;
- нижний затемненный overlay;
- название шаблона;
- mood-подпись.

Технический нюанс: live preview карточек обслуживает отдельный мини WebGL renderer, который рендерит только видимые карточки примерно в 20 FPS. Это сделано, чтобы не создавать WebGL-контекст на каждую карточку.

UX-дыра: список шаблонов длинный, особенно на mobile. Быстрые настройки и экспорт находятся после всей сетки шаблонов, а не рядом с выбором/результатом.

### Быстрые настройки

Блок `Быстрая настройка` содержит:

- `Цвет` — swatch row из 8 палитр, без `Свой`;
- `Скорость` — range `0..3`, step `0.01`, мапится на `state.speed`;
- `Движение` — range `0..9`, step `0.05`, мапится на `state.warp`;
- `Детализация` — range `0.3..5`, step `0.01`, мапится на `state.scale`;
- `Эффект` — сегменты `Нет`, `Пиксель`, `Полутон`, `ASCII`.

Эффекты мапятся так:

- `Нет`: `screen=0`, `dots=0`, `pixel=1`;
- `Пиксель`: `screen=0`, `dots=0`, `pixel=10`;
- `Полутон`: `screen=0`, `dots=1`, `pixel=4`;
- `ASCII`: `screen=2`, `dots=0`, `pixel=8`.

UX-нюанс: quick controls и Pro controls синхронизируют один и тот же `state`, но называют часть параметров по-разному:

- Simple: `Движение`;
- Pro: `Искажение`;
- Simple: `Детализация`;
- Pro: `Масштаб`.

Это полезно для разных аудиторий, но может сбивать, если пользователь переключается между режимами.

## 10. Pro-сценарий

Текущий путь Pro:

```text
Перейти в Профи
-> выбрать формат/ratio
-> при необходимости включить бесшовность
-> выбрать тип поля
-> настроить скорость/масштаб/искажение/зерно
-> выбрать фактуру
-> настроить пикселизацию/полутон
-> выбрать палитру или кастомные цвета
-> экспортировать или открыть блок для сайта
```

### 10.1. Размер

Форматы:

- `9:16`;
- `4:5`;
- `1:1`;
- `5:4`;
- `3:2`;
- `16:9`.

Переключение формата меняет preview и экспортные размеры.

Дополнительно:

- переключатель `Бесшовный паттерн`;
- если включен, появляется slider `Плиток в ряд`, range `2..10`, step `1`, default `5`;
- canvas wrap получает класс `seamless`.

UX-нюанс: категория `Паттерн` в Simple ставит ratio `1:1`, но не включает `seamless` автоматически, хотя подсказка говорит `Бесшовная плитка — повторяется без швов`. Дефолтный шаблон `Соты` тоже имеет `seamless=0`. Это расхождение между обещанием категории и фактическим состоянием.

### 10.2. Поле

Доступно 10 типов поля:

- Поток;
- Течение;
- Клетки;
- Гироид;
- Труше;
- Интерф.;
- Калейдо;
- Линии;
- Сетка;
- Золото.

Слайдеры:

- `Скорость`: `0..3`, step `0.01`;
- `Масштаб`: `0.3..5`, step `0.01`;
- `Искажение`: `0..9`, step `0.05`;
- `Зерно`: `0..0.3`, step `0.005`.

### 10.3. Фактура

Surface segment:

- `Квадрат`;
- `Гекс`;
- `ASCII`.

Слайдер:

- `Пикселизация / ячейка`: `1..40`, step `1`.

Дополнительно:

- `Полутон` доступен только при `Квадрат`;
- если полутон включен, появляется `Размер точки`: `4..30`, step `1`.

Технический нюанс: pixel slider всегда доступен. Для square это пикселизация, для hex/ascii — размер ячейки.

UX-нюанс: название `Квадрат` неочевидно как режим без фактуры/основной surface. В Simple пользователь видит эффект `Нет`, а в Pro видит surface `Квадрат`.

### 10.4. Цвета

Палитры:

- Сияние;
- Закат;
- Океан;
- Сумерки;
- Уголь;
- Мята;
- Ирис;
- Хром;
- Свой.

Под палитрами расположены 4 color input, соответствующие четырем stop-цветам градиента.

Поведение:

- выбор готовой палитры ставит `state.pal` и копирует stop colors;
- изменение любого color input переводит палитру в `Свой`;
- в Simple быстрая строка цветов показывает все палитры, кроме `Свой`;
- Pro позволяет редактировать 4 конкретных stop colors.

UX-нюанс: у пользователя есть сразу три цветовые сущности:

- тема UI;
- палитра фона;
- кастомные stop colors.

Сейчас они находятся в разных местах, но визуально похожи.

## 11. Экспорт

Блок называется `Скачать`. Он виден в Simple и Pro.

Настройки:

- `Разрешение (PNG и видео)`:
  - `Эскиз (640px)`;
  - `1080p (Full HD)`;
  - `2048×2048`;
  - `4K (Ultra HD)`.
- кнопка `Скачать PNG`;
- `Длительность видео`:
  - `Без лимита (стоп вручную)`;
  - `5 секунд`;
  - `10 секунд`;
  - `15 секунд`;
  - `30 секунд`;
  - `60 секунд`.
- кнопка `Записать видео`, подпись зависит от поддерживаемого формата: MP4 или WebM.

Логика PNG:

- берет выбранное export resolution;
- временно меняет renderer size;
- рендерит кадр;
- скачивает `zhizha_<timestamp>.png`;
- возвращает preview size.

Логика видео:

- выбирает MIME из списка MP4/WebM;
- если MediaRecorder недоступен, кнопка disabled;
- запись идет через `canvas.captureStream(60)`;
- bitrate `16_000_000`;
- при старте запись переводит сцену в live, если она была не live;
- повторный клик останавливает и скачивает файл;
- во время записи показывается `recDot` с таймером;
- selector длительности блокируется на время записи.

UX-дыры:

- Для creator-пользователя первым идет PNG, даже если выбран TikTok/Reels и подсказка говорит, что видео важно.
- Подсказка содержит технические детали: MP4/WebM, canvas, GPU, 4K. Это полезно, но слишком рано для основного сценария.
- Нет явных platform export presets рядом с категориями: категория меняет ratio, но export resolution остается абстрактным.
- Нет предпросмотра фактических пиксельных размеров перед скачиванием.
- Нет явного состояния успеха после скачивания, кроме нативного download.

## 12. Share, import и embed внутри приложения

Блок находится только в Pro:

`</> Для сайта · ссылка и встраивание`

Внутри:

- `Ссылка на эту сцену`;
- `Код для встраивания`;
- textarea для iframe-кода, скрыта до генерации;
- input `Вставь ссылку ЖИЖА (#s=…)`;
- `Загрузить сцену из ссылки`;
- hint с объяснением `#s=`.

### Ссылка сцены

Кнопка кодирует state в base64 JSON и пытается скопировать URL в clipboard.

Если clipboard недоступен:

- меняет текущий address bar на `#s=...`;
- показывает toast `Ссылка в адресной строке`.

### Embed code

Кнопка генерирует:

```html
<iframe src="..." style="border:0;width:100%;height:100%;aspect-ratio:..." loading="lazy" allow="fullscreen"></iframe>
```

Затем:

- показывает textarea;
- вставляет туда код;
- фокусирует и выделяет textarea;
- пытается скопировать код в clipboard;
- показывает toast.

UX-нюанс: внутри приложения генерируется только iframe-код. При этом в `embed/EMBED.md` основной рекомендуемый способ — `<zhizha-bg>` web component. Пользователь из UI не узнает про лучший способ подключения.

### Import

Input принимает ссылку с `#s=...`. Если ссылка валидна:

- декодирует state;
- обновляет все UI controls;
- пересчитывает uniforms и resize;
- показывает toast `Готово. Сцена на месте`.

Если невалидна:

- toast `Не разобрал ссылку — проверь её`.

## 13. State model

Основной state:

```js
{
  speed: 0.6,
  scale: 1.6,
  warp: 4.5,
  grain: 0.06,
  field: 0,
  pal: 0,
  seamless: 0,
  tile: 5,
  screen: 0,
  pixel: 1,
  dots: 0,
  dot: 10,
  cols: [...],
  seed: random,
  ratio: '16:9'
}
```

URL serialization keys:

- `sp` speed;
- `sc` scale;
- `wa` warp;
- `gr` grain;
- `fl` field;
- `pa` palette;
- `sm` seamless;
- `ti` tile;
- `scr` screen;
- `px` pixel;
- `dt` dots;
- `dz` dot;
- `sd` seed;
- `ar` ratio;
- `co` custom colors.

Decode behavior:

- если ratio невалиден, fallback `16:9`;
- если декодирование падает, возвращает `false`;
- после успешного import снимает active class со старых preset chips, но DOM-блока `#presets` сейчас нет.

## 14. Rendering и performance-поведение

Главный renderer:

- `THREE.WebGLRenderer`;
- `preserveDrawingBuffer:true` для экспорта;
- `powerPreference:'high-performance'`;
- DPR ограничен `1.5`.

Render loop:

- capped примерно на 60 FPS;
- рисует, если live, ripple active или state dirty;
- при hidden tab останавливает requestAnimationFrame;
- при возвращении вкладки продолжает.

Reduced motion:

- если `prefers-reduced-motion: reduce`, `live` стартует false;
- CSS transitions/animations отключаются через media query.

Pointer interaction:

- canvas получает role `img` и aria-label;
- pointer/touch movement меняет uniforms mouse и создает локальную рябь.

FPS:

- desktop показывает `FPS` в правом нижнем углу;
- mobile FPS скрыт.

UX-нюанс: live/static toggle в UI убран, но в коде осталась логика `setMode(l)`. Пользователь не управляет паузой напрямую.

## 15. Accessibility inventory

Что уже есть:

- основной canvas имеет `role="img"` и описание;
- toast имеет `role="status"` и `aria-live="polite"`;
- mode segment использует `aria-pressed`;
- ratio/surface buttons используют `aria-pressed`;
- switches имеют `role="switch"`, `tabindex="0"`, `aria-checked`, `aria-labelledby`;
- swatches и palette chips доступны с клавиатуры через Enter/Space;
- focus-visible стили есть для кнопок, switches, inputs, selects;
- reduced motion частично учитывается;
- iframe web component скрыт от accessibility tree через `aria-hidden`.

Риски:

- В `themePop` стоит `role="menu"`, но вложенные theme buttons не имеют role `menuitem`; по поведению это скорее popover/radio group.
- Некоторые элементы-«кнопки» сделаны `div role=button`, хотя можно использовать native button.
- Template cards имеют aria-label `Шаблон: ...`, но mood-подпись может быть недоступна как отдельная информация.
- Color swatches в Simple имеют только `title` и `aria-label`, но нет текстовой подписи рядом; для зрячих пользователей названия палитр не видны.
- На mobile длинный список шаблонов до настроек создает большую клавиатурную дистанцию.
- Технические подсказки мелким текстом могут быть низкоконтрастными на части тем.
- Нет явной проверки contrast всех тем и состояний на WCAG.
- Нет полноценной проверки screen reader flow: аудит основан на DOM/code/visual inspection, не на тесте с VoiceOver.

## 16. Текущие UX-сильные стороны

- Первый экран сразу показывает результат: нет лендинга, регистрации или пустого состояния.
- Creator-модель уже существует: категории, шаблоны, quick controls.
- Preview большой и всегда рядом с настройками на desktop.
- На mobile preview закреплен сверху, пользователь не теряет визуальный результат.
- Шаблоны имеют живые превью, а не только текст.
- Категории автоматически меняют ratio.
- Simple и Pro используют один state, поэтому изменения не теряются при переключении.
- Export и embed работают без backend.
- URL сцены переносит почти весь state.
- Есть понятный fallback для WebGL error.
- Есть embed mode без UI.

## 17. UX-дыры и противоречия

### 17.1. Главный пользовательский путь длиннее, чем обещает продукт

Слоган говорит `живые фоны за 20 секунд`, но на mobile пользователь видит длинную сетку шаблонов до quick controls и экспорта. В TikTok/Reels список включает 38 карточек.

### 17.2. Категория задает формат, но не всегда задает ожидаемое действие

Примеры:

- TikTok/Reels подсказывает, что видео на первом месте, но export block первым показывает PNG.
- Паттерн подсказывает бесшовность, но seamless не включается автоматически.
- Обложка говорит про PNG, но export block не меняет приоритет actions.

### 17.3. Simple и Pro используют разные языки для одних параметров

Simple:

- Движение;
- Детализация;
- Эффект.

Pro:

- Искажение;
- Масштаб;
- Фактура;
- Пикселизация / ячейка.

Это оправдано, но без bridge-подсказки пользователь может не понять, куда исчезли его настройки.

### 17.4. Export не адаптирован к выбранной категории

Сейчас category меняет ratio и hint, но не перестраивает export panel. Нет creator-specific export affordances:

- видео для TikTok/Reels/Stories;
- PNG для обложки;
- 16:9 для презентации/YouTube;
- tile export для паттерна.

### 17.5. Цветовые настройки конкурируют друг с другом

Есть:

- тема оформления;
- палитра сцены;
- кастомные stop colors.

Все выглядят как цветовые свотчи/градиенты. Тексты объясняют разницу, но разделение в mental model слабое.

### 17.6. Developer/embed-функции разорваны

В UI можно скопировать iframe. В документации главный способ — `<zhizha-bg>`. В UI нет выбора: iframe vs web component vs fallback.

### 17.7. Legacy/documentation mismatch

README говорит про 16 шаблонов, а текущий Simple UI показывает 52 creator-шаблона. В коде `PRESETS` действительно содержит 16 старых пресетов, но DOM-элемента `#presets` нет, поэтому старый блок не отображается.

`docs/CREATOR_UI_IMPLEMENTATION_PROMPT.md` упоминает `embed/fluid-bg.js`, но в текущем проекте файл называется `embed/zhizha-bg.js`.

### 17.8. Embed hash nuance

Если пользователь на уже открытой странице вручную добавит `#embed=1`, UI не исчезнет до перезагрузки, потому что embed mode проверяется только в `init()`.

### 17.9. Нет сохранения пользовательской сцены как workspace/history

Сцена кодируется в URL, но нет:

- недавних сцен;
- избранных;
- сохранения в localStorage;
- истории изменений;
- undo/redo.

Это не баг текущей версии, но важное ограничение UX.

### 17.10. Нет явной обратной связи по некоторым действиям

Есть toast для share/embed/import. Но для PNG/video download пользователь полагается на browser download behavior. Нет внутреннего состояния `готово`, `скачивание началось`, `ошибка скачивания`.

## 18. Текущие сценарии пользователя

### 18.1. Сценарий: creator делает фон для Reels

1. Открывает приложение.
2. Видит выбранную категорию TikTok/Reels.
3. Видит вертикальный live preview.
4. Выбирает шаблон из большой сетки.
5. При необходимости фильтрует по mood.
6. Скроллит до quick controls.
7. Настраивает цвет, скорость, движение, детализацию, эффект.
8. Скроллит до `Скачать`.
9. Выбирает разрешение.
10. Выбирает длительность.
11. Нажимает запись видео.
12. Нажимает стоп или ждет лимит.
13. Получает файл.

Главная friction-точка: быстрые настройки и экспорт находятся после шаблонов, а не всегда видны рядом с preview.

### 18.2. Сценарий: пользователь делает статичную обложку

1. Выбирает категорию `Обложка`.
2. Ratio становится `4:5`.
3. Применяется `Фейерверк`.
4. Выбирает шаблон.
5. Настраивает быстрые параметры.
6. В export block скачивает PNG.

Friction: export block не показывает, что PNG теперь главный recommended action; это видно только из общей структуры, где PNG стоит первым всегда.

### 18.3. Сценарий: пользователь делает фон под текст

1. Выбирает `Фон под текст`.
2. Ratio становится `16:9`.
3. Применяется `Затишье`.
4. Доступны calm/deep/mono шаблоны.
5. Подсказка обещает читаемый центр и мягкое движение.

Friction: UI не показывает overlay/текстовую проверку читаемости. Пользователь должен на глаз оценить, подойдет ли фон под текст.

### 18.4. Сценарий: пользователь делает паттерн

1. Выбирает `Паттерн`.
2. Ratio становится `1:1`.
3. Применяется `Соты`.
4. Подсказка говорит про бесшовную плитку.
5. Чтобы включить бесшовность, пользователь должен перейти в Pro и включить `Бесшовный паттерн`.

Friction: обещание `Паттерн` не совпадает с доступностью настройки в Simple.

### 18.5. Сценарий: designer/developer берет фон на сайт

1. Создает сцену.
2. Переходит в `Профи`.
3. Скроллит до `Для сайта`.
4. Открывает details.
5. Копирует ссылку сцены или iframe.
6. При необходимости читает `embed/EMBED.md`, чтобы использовать `<zhizha-bg>`.

Friction: recommended web component не доступен из UI.

### 18.6. Сценарий: пользователь получает ссылку на сцену

1. Открывает URL с `#s=...`.
2. Приложение декодирует state.
3. Если `#s` валиден, стартовая creator-категория не выбирается.
4. UI обновляет controls.
5. Активная карточка шаблона может не быть отмечена, потому что сцена может не соответствовать конкретному шаблону.

Friction: пользователь видит сцену, но может не понимать, из какой категории/шаблона она была сделана.

## 20. Что Claude Design должен понять перед редизайном

Это не просто генератор абстрактных градиентов. Это редактор живого фона с двумя аудиториями:

- creator, которому нужно быстро получить фон для видео/сторис/обложки;
- advanced/developer user, которому нужны точные параметры, URL state и embed.

Нынешняя логика уже частично creator-first, но UX все еще смешивает:

- подбор результата;
- техническую настройку процедурного поля;
- экспорт медиа;
- developer embed;
- тему интерфейса;
- палитру сцены.

Самые важные сущности, которые нельзя потерять:

- live preview как главный герой;
- category -> template -> quick tune -> export;
- Pro как вторичный слой;
- URL scene sharing;
- PNG/video export;
- embed mode;
- web component;
- mobile sticky preview.

Самые важные проблемы, которые нужно учитывать:

- длинная лента шаблонов отодвигает настройку и экспорт;
- export не адаптируется к выбранному use case;
- Simple и Pro используют разный язык без явного bridge;
- pattern category не включает seamless;
- website embed в UI слабее, чем embed docs;
- тема UI и палитра сцены могут путаться;
- mobile основной путь требует слишком много скролла.

## 21. Ограничения аудита

- Аудит не проверял реальную запись видео во всех браузерах и кодеках.
- Аудит не проверял все export-файлы визуально после скачивания.
- Аудит не проводил полноценный screen reader test.
- Аудит не замерял производительность на слабых GPU.
- Аудит не менял текущий код продукта, кроме добавления этого документа и папки со скриншотами.
